RAG：

1、你这个RAG项目如何来进行评测？

> 项目评测的核心目标是生成的测试用例是否覆盖 PRD 里的功能点、业务规则、异常场景、边界条件、权限规则、状态流转和资损风险点，并且每条用例能追溯到对应的需求或知识来源。
>
> 分阶段评估指标
>
> 首先是查询改写和拆分阶段，对于改写主要是看改写的query有没有偏离原始需求，就是改写准确率。因此这个里面评测集合就是用户原始的query和改写后的query，人工或 LLM Judge 判断有没有偏离原意（比如用户就输入签约失败的测试用例，就要补充从首次支付签约失败，续费签约失败，支付失败之后重试签约失败等等。但是要是扩展成签约成功就偏离原意了），对于拆分主要是看拆分完整率（是不是把prd所有的功能点都拆分成子任务了）、拆分的准确率（正确子任务数 / 系统拆分出的子任务总数）等等
>
> 下面是意图识别和检索，意图识别就是找每个子任务对应的意图，这和检索是相似的，检索是找每个子任务的chunk。因此可以用召回率、准确率、命中率和平均倒数排名来进行衡量。但是这两个还是有区别的，意图识别是交给模型来做的，检索则是通过余弦相似度和BM25去检索的。
>
> 在下面是重排序阶段评估，重排序的目的是把更符合的chunk排在前面，因此要计算重排序前后的MRR和召回率还有噪声下降率（相关性不大的文档是不是被排到后面去了）
>
> 最后是生成阶段评估，生成出来的测试用例到底好不好？从各种覆盖度的角度来看、从用例有效性来看。其次是忠实度（模型生成的答案是否忠实于检索到的 chunk 内容，有没有编出 chunk 里没有的信息）和答案相关性（模型生成的答案是否回答了用户的问题？有没有答非所问？）。对于忠实度，用 LLM 对每条生成的测试用例的忠实度打分（1~5 分，5 分表示完全忠实，1分表示严重编造）对于答案相关性，也是用LLM对用每条生成的测试用例的相关性打分。
