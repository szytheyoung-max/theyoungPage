# Feature产品配额

Feature 有哪些字段：feature含义表示产品的配额

- Feature code：自己定义的唯一的feature编码
- Feature 名称：自己定义的feature名称
- Feature 显示名称：自己定义的，展示给用户的feature名称
- pid：勾选有接入该Feature功能点的PID。点击选择产品，可多选有接入该Feature的产品
- 授权计量单位：如同时在线设备数/消耗品设备数等
- 是否允许超量：允许超量，表示该功能点去售卖时，允许用户超量使用，如云存量、如消耗品的使用次数。勾选允许超量之后，需要在创建SKU售卖时，为超量授权配置超量单价，用于超量账单使用

#  Feature的售卖单位

售卖单位有如下字段：

- 售卖单位标识：自己定位售卖单位唯一标识
- 权限计量维度（限制单位）：Feature的**授权计量单位**，如同时在线设备数/消耗品设备数等。详见https://confluence.300624.cn/pages/viewpage.action?pageId=436833646
- 售卖单位：
  - 为什么需要售卖单位：因限制单位一般会到计量的最小颗粒度，如云存储当前售卖单位是1G，100G，以G/T来售卖，但实际上授权计算的计量是 字节，导致2边值不一致，因此增加售卖单位。售卖单位是依赖于限制单位基础上进行乘法计算的。
  - **售卖单位的显示名称以及多语言**：用于购物车以及用户中心显示使用。
  - 需配置1售卖单位=N*限制单位
  - 限制单位和售卖单位是1对多关系，同一个限制单位可创建多个售卖单位
  - 是否规律性发放：主要针对消耗品类，每天/月/季发放，表示每天/月/季会重置用户的授权单位值，例如：售卖1G/月（每月发放）的Feature，该Feature售卖按年，表示在用户这一年内，每个月都给用户发1G的流量。
- 售卖单位展示名称：展示给用户看的名称
- 售卖单位换算系数：就是售卖单位里面N

# 订阅模板

订阅模板也是在UMS创建的，订阅模板实际上就是会员方案

- 从用户视角，订阅模板是权益包/会员等级，如：Filmstock基础会员、 Filmstock高级会员、云存储黄金会员，不同会员不同权益
- 从系统视角，订阅模板是一系列Feature Code组成的权益组合，包含权益类型、数量、发放方式等属性；订阅模板**没有时长的概念**，时长都在SKU上添加，订阅模板只有**订阅收费类型**：自动续费的（如按年自动续费） or 一次性非终身的（如单卖1年）or 一次性终身的。

- 为了兼容旧的用户订阅数据，所有在CART4中创建的SKU，都会自动创建一份订阅模板。

## 售卖设置

### 订阅会员

售卖模式：订阅会员（周期性付费），定期自动续费，只要用户不主动取消订阅，就会自动续费

售卖类型：

- 订阅叠加时长：在原有权益上**延长有效期**。例如，会员原定下月到期，再次购买后将延长到期时间。比如说购买了1年的喵影会员，从2020.1-2021.1。再次期间又买了3个月喵影会员，所以现在就是从2020.1-2021.4
- 订阅叠加数量：在原有权益上**增加可用次数**，原有效期不变。例如，原可使用 1 台设备，再次购买后可增加更多设备数

是否是免费会员

- 勾选【是】后，请用该会员方案申请兑换码（兑换密钥形式），用于发放免费会员权益。用户登录后，**产品侧自动完成兑换**，用户可直接获得免费权益（无感知过程）。当用户购买 VIP 会员后，VIP权益将覆盖免费会员权益，**两者不会叠加**

### 买断会员

售卖模式：买断会员（用户一次性付清费用后，在约定的有效期内（如2年/ 永久）持续享受会员服务， **后续不会再自动扣费**）

售卖类型：

- 限时买断：一次性付费，在指定时间内享受权益
- 永久买断：一次性付费，永久享受权益

叠加方式：

- 叠加时长：在原有权益上**延长有效期**。例如，会员原定下月到期，再次购买后将延长到期时间
- 叠加数量：在原有权益上**增加可用次数**，原有效期不变。例如，原可使用 1 台设备，再次购买后可增加更多设备数

### 用量包

售卖模式：针对 AI 积分、特定功能或云存储空间等的独立加购包，用于补充使用额度。支持一次性购买或自动续订。

售卖类型：

- 订阅叠加时长：同上
- 订阅叠加数量：同上
- 限时买断：同上
- 永久买断：同上

## 授权规则设置

授权类型：主要分为席位数和权益。席位数包含了同时在线设备的个数、账号授权。权益包含了某个功能的使用次数、某个增强的功能服务、云存储的容量大小等等

然后要明确配额规则，对于权益-增强服务的规则就是增强的哪个服务。对于权益-功能次数的规则就是次数

然后要明确配额的数量

然后要明确配额发放的规则：按天、按季度、按月、一次性、按年等等

然后要明确配额可调：如果该会员方案计划参与阶梯定价或线下灵活兑换授权，请将配额可调设为【是】。实际授权的配额数量，将根据用户购买 SKU 时的选择或线下授权填写的数量确定

接下来是免费使用的配额和低价体验的配额

最后是配额标识配置，就是和feature绑定，并填写feature里面的配额售卖单位

## 分组设置

方案分组：对会员方案进行分组，是实现**购买互斥**或**授权叠加时长**的重要条件之一，但需与订阅叠加时长或限时买断叠加时长及方案等级条件配合使用，才能生效

- **购买互斥示例**
  - 同一分组中有三个不同等级的会员方案 A、B、C，售卖类型均为订阅叠加时长
  - 若用户已购买会员方案 A，再尝试购买方案 B，将无法直接购买（发生互斥）
  - 用户只能通过会员方案升降级的方式，获得方案 B 的权益

- **授权叠加时长示例**
  - 同一分组中包含会员方案 A、B、C，售卖类型均为订阅叠加时长或限时买断叠加时长
  - 其中 A 与 B 等级相同且权益维度一致（产品ID+配额规则一致），C 的等级不同
  - 若用户已购买 A，再购买 B，则会员**有效期将延长**
  - 若用户已购买 A，再购买 C，则无法直接购买，或同时拥有 A 和 C 的权益。（PS：若A、C均为订阅叠加时长售卖类型，则购买互斥）

方案等级划分：会员权益可设置等级（如基础版、高级版、尊享版等），方案等级是实现**授权叠加时长**的重要条件之一，需与订阅叠加时长或限时买断叠加时长及相同方案分组条件搭配使用，才能生效

- 同一分组中包含会员方案 A、B、C，售卖类型均为订阅叠加时长或限时买断叠加时长

- 其中 A 与 B 等级相同且权益维度一致（产品ID+配额规则一致），C 的等级不同

- 若用户已购买 A，再购买 B，则会员**有效期将延长**

- 若用户已购买 A，再购买 C，则无法直接购买，或同时拥有 A 和 C 的权益。（PS：若A、C均为订阅叠加时长售卖类型，则购买互斥）

方案归属业务线：选择一个业务线即可

## 50购物车和40购物车的异同

当前系统里面同时存在40购物车和50购物车对应的SKU，而且为了兼容使用，当前有以下注意事项：

- Cart40里面（TMS-40商品中心-40SKU管理）创建的SKU，保存后需要到TMS-40商品中心-价格管理页面里面去做审批，提交审批或自动发起OA流程（流程在发起者的兴家OA流程里面）；
- 步骤1中**Cart40 SKU** OA审核通过后这份SKU会**自动同步到50SKU**列表里面（TMS-50商品中心-50SKU管理），Cart50里面创建的SKU不会同步到Cart40里面去，而且40同步到50里面的SKU，对应的**购物车**APP需要**重新选择**保存一次才能使用50里面对应的购买链接；
- Cart40购物车和50 购物车价格配置差异：
  - 40SKU是基于**PID+License**来生成的，所以需要保证PID+License组合的**唯一性**，PID和License需要提前在PMS和UMS里面创建好才能使用，**40SKU的周期是在License里面控制**的；
  - **50SKU**是基于**订阅模板，SKU**里面可以控制**周期**，需要提前在UMS里面创建好订阅模板，**同一个订阅模板可以创建多个SKU**；
  - 40SKU只有**固定价格**；
  - **50SKU**可以有**固定价格**，也可以实现**灵活定价（阶梯定价、区间定价、固定+阶梯定价、固定+区间定价）**；

- Cart40 SKU的购买链接从TMS-价格中心-价格管理页面里面获得，有基础购买链接、coupon购买链接；Cart50 SKU的相关购买链接从TMS-商品中心-SKU管理页面里面获得（基础购买链接、活动购买链接、买赠活动购买链接）；

- **Cart40** SKU使用**coupon**可以实现打折，**coupon链接**或者**coupon码**的方式，但是请注意，这个coupon是大家**公用**的，没有和用户做绑定，任何人拿到这个coupon链接或coupon码都能购买对应配置的关联SKU商品；
- **Cart50** SKU里面当前可以配置使用**公共优惠券和私有优惠券**；私有**优惠券**需要**用户领用**或者**后台派发**给特定的用户后，实现**和WSID绑定**得到具体唯一的优惠券码，每个人一个码不能被别人公用；

请注意：以前老的做法：为了兼容40购物车里面的coupon码，也可以人为的让研发把40里面的coupon码同步到50 优惠券里面去使用，也能实现这个优惠券码被很多人都拿到共同使用，同步过去的优惠码可以直接关联50里面的SKU；

- 要实现**买A送B**的功能，40购物车里面通过**组合SKU**的方式来实现，50购物车里面可以通过配置赠品SKU然后走**买赠活动**（赠送其他SKU）来实现，50里面也可以通过**拼接additional_sku**来来实现组合购买，不过价格是SKU的售价相加得到，additional_sku上面也可以参与活动来实现打折售卖；

- 50购物车里面有**活动**管理功能，可以定义更丰富的活动（**普通活动打折/抵扣**，能**重定义周期和价格的折扣续订活动、一次付活动、免费试用活动**）来促进营销，而40购物车里面不行；

- 当前所有**移动端的关联映射SKU，可以使用50的SKU**；

**关于TMS里面SKU对应购买链接里面的购物车域名获取规则：**

1. 40单品SKU的购买链接是根据品牌+pid在PMS里面配置的产品购买页的域名后缀拼起来的；
2. 40组合SKU的购买链接是获取的TMS里面选择的购物车app里面的域名；
3. 50SKU的购买链接都是获取的TMS里面选择的购物车app里面的域名；

### 价格设置

SKU里面支持SKU按币种定价、同一个币种下面按国家地区分区定价；

SKU获取取价格的过程，客户端或者购买页会根据用户的IP判断所在的国家，然后按照下面的步骤来获取对应的币种-价格数据进行展示：

1. 先去已经支持本地化的国家-币种列表中寻找对应的本地币种的定价，如果匹配到了，那么就直接给请求端返回并展示对应国家的本地币种和定价；
2. 如果1中匹配不到，那么就在USD和EUR里面去寻找是否有特定国家地区的分区定价（USD优先），如果匹配到了，那么就直接给请求端返回并展示USD或EUR币种中对应的国家分区定价；
3. 如果1和2都匹配不到，那么就使用USD和EUR的通用价格（USD优先），也就是给请求端返回并展示default_USD/default_EUR的定价；
4. 如果1、2和3都匹配不到，也就是说SKU里面本地币种定价也没有配置USD和EUR的币种定价，那么就获取已配置的第一个币种里面的通用价格返回；

#### 灵活定价

针对订阅模板里面设置“售卖单位值可变”为‘是’的feature code，可以进行每个售卖单位上的定价，分为以下几种类型：

##### 阶梯定价

根据不同阶梯之间不同价格，分别计算，最终得到总价，举例如下：

币种1：美元，选择阶梯定价。0-5之间每单位售卖5，6-10之间每单位售卖6，11-15之间每单位售卖7

若购买13个单位，则总价`5*5+5*6+5*7 = 76`

##### 区间定价

根据不同区间定不同单价，根据用户购买数量命中哪一区间，计算总价，举例如下：

币种1：美元，选择区间定价。0-5之间每单位售卖5，6-10之间每单位售卖6，11-15之间每单位售卖7

若购买13个单位，则总价`13*7 = 91`

##### 固定价格+区间价格

在区间定价基础上，多出一部分固定价格，示例如下：

币种1：美元，选择固定价格+区间价格。0-5之间每单位售卖5，6-10之间每单位售卖6，11-15之间每单位售卖7。固定价格为30。

若购买13个单位，则总价`50 + 13*7 = 141`

##### 固定价格+阶梯价格

在阶梯定价基础上，多出一部分固定价格，示例如下：

币种1：美元，选择固定价格+阶梯价格。0-5之间每单位售卖5，6-10之间每单位售卖6，11-15之间每单位售卖7。固定价格为30。

**使用门槛**：当前无实际用途；

**优惠内容**：

- **抵扣券**：此优惠券**减免的金额**，例如某订单为10美元，优惠金额为2美元，则最终价格为8美元-

**每人限领次数**：一个WSID限制领取的次数，数量填写-1代表无限，每一次领券后会生成一个优惠券实例（唯一的优惠券码）和WSID进行绑定；

**公开设置**：当前无实际用途；
